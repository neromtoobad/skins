## Complexity Assessment
Complexity: Complex
Rationale: Greenfield build of a full MCP server subsystem spanning ~12 source files, three independent input pipelines (text/URL/image), a templated React/Framer Motion component generator, an LLM client with fallback heuristics, a verification harness, and a self-contained HTML preview — all with strict quality bars on output validity.

## Goal
Build `skins-mcp`, a Model Context Protocol server that converts a natural-language vibe (or a scraped URL or a base64 image) into a complete, production-ready React + Tailwind + Framer Motion design system, exposed as three MCP tools that return tokens, components, a full page layout, a self-contained HTML preview, and copy-paste `.tsx` files.

## Context
- The workspace `/home/harness/cyops_data/workspace/skins` currently contains only a `.git/` and a `.gitignore` (no source code yet). This is a greenfield build.
- The product is a TypeScript MCP server published as the npm package `skins-mcp`, with entry point `src/index.ts` and a `demo.ts` driver.
- Output quality bar: valid TypeScript React, real Tailwind utility classes, correct Framer Motion API usage (`motion.div`, `variants`, `initial`/`animate`/`exit`), and a self-contained HTML preview that renders without a build step.
- Three input modes must work — `generate_from_vibe` (text), `generate_from_url` (fetch + cheerio), `generate_from_image` (base64 → dominant colors) — each returning the same five-output shape.
- Zero required API keys for `from_vibe`: a built-in heuristic engine of hardcoded presets must cover common vibes. LLM is invoked only as a refinement when no preset matches the keywords.
- LLM client must be configurable against any OpenAI-compatible endpoint via `OPENAI_API_KEY` + `OPENAI_BASE_URL` + `OPENAI_MODEL` env vars; absence gracefully degrades to preset-only behavior.
- Runtime: Node.js ≥ 18, TypeScript executed via `ts-node` for the demo and via the MCP SDK's stdio transport for the server.
- This is a backend / CLI deliverable, not a hosted web app, so Vercel/Cloudflare Pages deployment does not apply.

## Assumptions
- Node 18+ is available in the container; the implementer may run `npm install` and `npx ts-node` without additional system packages.
- The MCP server uses **stdio transport only** (the de-facto MCP standard for local tool servers); no HTTP/SSE listener is required.
- LLM environment variables follow the OpenAI convention: `OPENAI_API_KEY`, `OPENAI_BASE_URL` (default `https://api.openai.com/v1`), `OPENAI_MODEL` (default `gpt-4o-mini`). When unset, the LLM path throws a typed `LlmUnavailableError` and the call falls back to the closest preset.
- Color extraction in image mode uses the `sharp` package plus a small in-house k-means-lite (k=5) color quantizer — no heavy ML model.
- The HTML preview uses the **Tailwind Play CDN** (`<script src="https://cdn.tailwindcss.com"></script>`) plus inline `<style>` for the token-derived CSS custom properties. The file is fully self-contained (no local asset references).
- The `demo.ts` script writes outputs into `demo-output/` at the workspace root, one folder per mode (`demo-output/vibe/`, `demo-output/url/`, `demo-output/image/`).
- "Self-contained HTML preview" is interpreted as a single `.html` file that can be opened directly in a browser (file://) without a build step. CDN scripts are permitted; no local file imports are required.
- Browser-based verification (open the preview, inspect screenshot) uses the preinstalled `agent-browser` CLI per the harness rules, and any raw reference assets (HTML snapshots, screenshots) are saved under `docs/references/preview-snapshots/`.

## Implementation Notes

### Architecture
A single canonical `DesignTokens` object flows through every generator. Input modes feed an `Aesthetic Interpreter` that returns tokens; the same tokens then drive deterministic, template-based generators for components, layout, and HTML preview. The LLM is touched only inside `generateTokens` to translate a free-form vibe into the token shape; component/layout/preview code is never LLM-generated, which guarantees the Tailwind/Framer Motion quality bar.

### Key dependencies
- `@modelcontextprotocol/sdk` (latest 1.x) — provides `McpServer`, `StdioServerTransport`, and `zod`-backed tool schemas.
- `cheerio` (1.x) — HTML parsing for URL mode.
- `sharp` (0.33.x) — image decoding and resizing for image mode.
- `openai` (4.x) — OpenAI-compatible client; configured with `baseURL` from `OPENAI_BASE_URL`.
- `zod` (3.x) — schema validation for LLM responses and MCP tool input.

### Sequencing
1. **Scaffold + types** (AC-1, AC-2): `package.json`, `tsconfig.json`, `src/types.ts`. This unblocks every other module.
2. **Presets** (AC-3): `src/vibes/presets.ts` with 8 hand-tuned entries. Establishes the design vocabulary.
3. **LLM client** (AC-12): `src/llm.ts` with the typed error class.
4. **Token generator** (AC-4): `src/generators/tokens.ts` — pure function, no MCP coupling. Easy to unit-verify.
5. **Component generator** (AC-5): `src/generators/components.ts` — token-driven template strings, one template per component, filled by a `fillTemplate()` helper. Add `src/generators/tailwind-dict.ts` enumerating allowed classes.
6. **Layout generator** (AC-6): `src/generators/layout.ts` — emits a `DemoPage` `.tsx` that imports the generated components.
7. **Preview generator** (AC-7): `src/generators/preview.ts` — emits HTML with the Tailwind Play CDN, a `<style>` block of CSS variables, and static markup mirroring the layout sections.
8. **Tool wrappers** (AC-8, AC-9, AC-10): three small files in `src/tools/`, each one a single `server.tool(name, schema, handler)` call.
9. **Server entry** (AC-11): `src/index.ts` wires the three tools and connects stdio transport.
10. **Demo driver** (AC-13): `demo.ts` plus a tiny `demo-assets/` generator that writes a sample PNG if missing.
11. **Verify script** (AC-15): `scripts/verify-output.mjs` runs Tailwind-class and Framer-Motion-API assertions.
12. **README** (AC-14): final documentation pass.
13. **Browser verification** (AC-16): open each generated `preview.html` with `agent-browser` and snapshot the result; save snapshots into `docs/references/preview-snapshots/`.

### Risk controls
- **LLM unreliability**: the LLM call is wrapped in a `try/catch` that falls back to the closest preset; the verify script never requires the LLM path to pass.
- **Tailwind class validity**: every class is built from a finite dictionary in `src/generators/tailwind-dict.ts` (an internal list of allowed classes per component variant). The verify script also runs a regex check.
- **Framer Motion API drift**: pinned to a single import `import { motion } from "framer-motion"` and the `variants`/`initial`/`animate` triple. The verify script greps for these tokens.
- **HTML preview rendering**: the preview is a static HTML+Tailwind document; any optional dynamic accent loaded from a CDN is wrapped in a try/catch so a CDN failure does not break the page.

### Deployment note
This deliverable is a local MCP server (Node CLI), not a hosted web application, so the Vercel/Cloudflare Pages one-click deploy tab does not apply. End users install it into Claude Desktop (or any MCP client) via `claude_desktop_config.json` as documented in the README.

## Out of Scope
- HTTP/SSE transport for the MCP server (stdio only).
- Authentication, rate limiting, or multi-tenant support.
- A web UI for SKINS itself (the product is an MCP server consumed by an MCP client).
- Live URL rendering of generated components (the user copies the `.tsx` files into their own app).
- Image generation or upscaling as part of the design system.
- Saving design systems to a persistent store — outputs are returned per call.
- More than the 5 documented components in v1 (no Modal, Drawer, Tabs, etc.).
- Internationalization of generated content.
- Tailwind v4 compatibility — pinned to v3.4+ for class-name stability.
